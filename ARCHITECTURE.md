# Architecture Overview

## Problem framing

The core challenge isn't "build a RAG pipeline" -- it's building one where
**citation accuracy is a first-class requirement** and **refusal is
structural, not aspirational**. Page numbers are attached to every chunk
at ingestion time and read back from retrieval results, never generated by
the model. Refusal is enforced by a relevance-score gate in code, backed up
by an exact refusal string the model is instructed to use verbatim when it
genuinely cannot answer from what it was given -- checked in code so
citations and the refusal message can never contradict each other.

## Pipeline

+-------+--------+   to unfiltered search on low confidence.

+----------------+   Dense (ChromaDB, BAAI/bge-small-en-v1.5) + Sparse

## Key design decisions

**Page-by-page PDF extraction with OCR fallback.**
`ingestion.py` extracts text one page at a time via PyMuPDF, tagging every
resulting chunk with its exact page number. Pages with no extractable text
(scanned/image-only pages) fall back to OCR (`ocr.py`, RapidOCR at 300 DPI)
rather than being silently skipped, so scanned content still becomes
searchable and citable.

**Sentence- and paragraph-aware chunking, not blind character slicing.**
`chunk_page_text` first splits on paragraph breaks (`\n\n`) and accumulates
whole paragraphs up to the chunk size limit. Only when a single paragraph
itself exceeds the limit does it fall back to `_split_long_text`, which
splits at sentence boundaries rather than cutting mid-word. This was a
direct fix after testing showed the original blind slicer producing
citation snippets that started mid-sentence or mid-word.

**Hybrid search (dense + BM25), not embeddings alone.**
Textbook questions often use exact terminology where keyword match
outperforms semantic similarity. BM25 catches these; dense search catches
paraphrased/conceptual questions. Reciprocal Rank Fusion combines the two
ranked lists without needing manually tuned score weights.

**Query rewriting for follow-up questions.**
Testing surfaced a real gap: a follow-up like "explain it in more detail"
carries almost no searchable signal on its own. `rewrite_query_node` runs
before routing and retrieval, using recent chat history to turn such
follow-ups into standalone questions or instructions. The sanity-check on
the rewrite deliberately does not require a question mark, since valid
rewrites of instructions are not phrased as questions.

**Router retries on weak results, not just empty results.**
Testing found a case where the router sent a question to the wrong subject
and the filtered search returned several chunks -- just weak, off-topic
ones -- so a naive "retry only if empty" safety net never triggered.
`retrieve_node` now retries unfiltered whenever the *best* score is still
below threshold, not just when nothing comes back at all.

**Cross-encoder reranking as a separate stage.**
A cross-encoder scores each (query, chunk) pair jointly for a more accurate
final ordering, run only on the fused shortlist rather than the whole corpus.

**Citations extracted from metadata, never asked of the LLM, and kept
consistent with the refusal path.**
Citations shown to the user are built directly from the retrieved chunks'
metadata, and explicitly cleared if the model's answer matches the exact
refusal string, so a refusal message can never appear alongside a
contradictory list of sources.

**Dockerized deployment with CPU-only PyTorch.**
The Dockerfile installs `torch` from PyTorch's dedicated CPU package index
before installing the rest of `requirements.txt`, rather than letting pip
resolve the default (CUDA-enabled) build. This avoids downloading several
gigabytes of unused GPU libraries, since standard Docker Desktop on macOS
has no GPU passthrough -- a concrete build-time optimization found and
fixed during testing, not a default choice.

## Known limitations / what I'd improve with more time

- **Ranking quality varies with how a topic is presented in the source
  text.** Testing confirmed this directly: searching the indexed corpus for
  "operating system" and "database" found the terms present (11 and 7
  chunks respectively), but only in incidental mentions -- e.g. "operating
  system" appearing in a textbook preface rather than in an actual
  definitional passage, and "database" appearing only in unrelated
  spreadsheet-saving instructions. In both cases the system correctly
  refused rather than fabricating a definition from these fragments -- the
  intended, safe behavior -- but it means the system can be conservative on
  topics not explicitly defined in the indexed source material, even when
  the term is technically present somewhere in the corpus.
- **The relevance threshold was tuned empirically** against real test
  queries (comparing scores on genuinely answerable vs. genuinely
  out-of-scope questions) rather than a formal labeled evaluation set.
- **No authentication/session persistence** -- out of scope for the
  assignment's core requirements; single-session in-memory chat history only.
- **No systematic performance optimization** -- e.g. caching repeated
  questions or reducing the number of LLM calls per query (currently up to
  4 per question: rewrite, route, generate, plus a separate streaming call
  in the UI). Scoped out given time constraints in favor of
  retrieval-quality work.

## Stack

- **LLM**: Groq (`openai/gpt-oss-120b`)
- **Embeddings**: `BAAI/bge-small-en-v1.5`
- **Vector store**: ChromaDB (persistent, local)
- **Sparse retrieval**: BM25 (`rank_bm25`)
- **Reranker**: `cross-encoder/ms-marco-MiniLM-L-6-v2`
- **OCR**: RapidOCR (onnxruntime)
- **Orchestration**: LangGraph (rewrite -> route -> retrieve -> generate)
- **UI**: Streamlit, with streaming responses
- **Deployment**: Dockerized (CPU-only PyTorch to keep image size reasonable
  without GPU acceleration)
